    - 아마존 프라임데이: AI 쇼핑 챗봇 '루퍼스' 도입 후 작년 대비 **11% 증가한 역대 최대 매출 142억 달러** 기록

**1-2. 기획 목적**

💡 보는 것을 넘어, 스스로 할 수 있게 (Autonomy)

- 정보 접근(Access)을 넘어선 '행위의 완성(Action)’
    - 기존의 보조 공학 기기(스크린 리더)는 정보를 **'읽어주는 것'**에 그쳤습니다.
    본 프로젝트는 읽어주는 것을 넘어, **'검색-비교-결제'라는 복잡한 쇼핑 프로세스를 AI 에이전트가 대신 수행(Action)**하여 시각장애인이 타인의 도움 없이 쇼핑을 완결짓게 하는 것을 목표로 합니다.
- 기술을 통한 사회적 비용 절감과 자립 지원
    - 활동 보조 비용 현황
        - **2025년 장애인활동지원 서비스**: 시간당 **16,620원** (디테일이슈, 2024.09.03)
        - **2025년 장애인활동지원 사업 예산**: **2조 5,323억 원** (한국장애인뉴스, 2025.01.24)
        - **2024년 현재 서비스 이용자**: **12.4만 명**
    - 핵심 메세지
        - 활동보조인이나 가족의 눈을 빌려야만 했던 쇼핑을 스스로 해결함으로써, 시각장애인의 **삶의 독립성(Independence)**을 확보하고 사회적 비용을 절감합니다.

**2.2 사용자 페르소나 (User Persona)** 

**- ex) 전맹 시각장애인**

- **이름/역할:** 김민수 (22세, 전맹 시각장애인)
- **Story:** 자취를 시작한 지 6개월 차. 생필품이 떨어질 때마다 쿠팡을 이용하려 하지만, '이미지'로만 구성된 상세페이지는 화면 낭독기가 읽어주지 않아 친구나 부모님께 매번 전화를 걸어 "이거 얼마야? 리뷰 좋아?"라고 물어봐야 함.
- **Needs:** 누구의 도움 없이 **말 한마디로** 최저가 생수를 찾고 결제까지 완료하는 '**완전한 쇼핑 독립**'.
- **Pain Points:** "이거 사줘"라고 하면 "어떤 옵션을 선택하시겠습니까?"라며 복잡한 선택지를 제시하는 기존 인터페이스의 불친절함.

### **3. 서비스 경험 설계 (Experience Design)**

**3.1 사용자 시나리오 (User Scenario)** 
1. **진입 단계:** 일상의 결핍을 인지하고 서비스를 호출

- **상황:** 늦은 밤, 민수는 내일 아침에 먹을 시리얼과 우유가 떨어진 것을 확인합니다. 기존 앱은 이미지 중심이라 혼자 주문하기 벅차다는 생각에 한숨을 쉽니다.
- **액션:** 민수는 데스크탑의 키보드를 통해 음성 인식 쇼핑 플랫폼인 (’ear shopping’)’을 입력해 서비스를 호출합니다.

2. **수행 단계:** 음성 대화형 쇼핑 및 최적의 조건 선택

- **민수:** "쿠팡에서 내일 아침에 올 수 있는 저지방 우유 1리터짜리 제일 싼 거 찾아줘."
- **서비스:** "쿠팡에서 로켓 배송 상품 3개를 찾았습니다. 가장 저렴한 건 A사 제품으로 2,900원이고, 100ml당 가격은 290원입니다. 평점이 가장 높은 건 B사 제품으로 3,200원입니다. 어떤 것을 장바구니에 담을까요?"

3. **완료 단계:** 배송 현황 확인 및 쇼핑 성공 경험 누적
- **확인:** 결제가 완료되면 서비스가 "주문이 완료되었습니다. 내일 오전 7시 전까지 문 앞으로 배송될 예정입니다."라고 안내하며 종료됩니다.

**5. 기술 설계 및 구현 계획 (Technical Strategy)
5.1 기술 스택 (Tech Stack)**
• **Database:**  (예: PostgreSQL, MongoDB 등)
• **Infrastructure:**  (예: AWS, Azure, Docker 등)

- **Front-end (접근성 최우선):** 사용자 인터페이스 구현 도구
    - **React.js** (컴포넌트 재사용성), **Next.js** (SEO 및 스크린 리더 최적화 추천).
    - **Voice Interface:** **Web Speech API** (브라우저 내장 STT/TTS). 별도 비용 없이 음성 인식/합성 가능.
        - [MDN Web Speech API 가이드](https://developer.mozilla.org/ko/docs/Web/API/Web_Speech_API)
    - **Accessibility (A11y):**
        - **Headless UI** or **Radix UI**: 접근성 표준(WAI-ARIA)이 이미 적용된 UI 라이브러리. (스크린 리더가 버튼인지, 탭인지 정확히 읽어줌)
        - [WAI-ARIA 공식 문서](https://developer.mozilla.org/ko/docs/Web/Accessibility/ARIA)
- **Back-end (안정성 & AI 연동) :** 서버 로직 및 API 처리
    - **Server:** **Spring Boot** (Java). 비즈니스 로직(주문, 결제, 회원관리) 담당.
    - **AI Serving:** **FastAPI** (Python). AI 모델(이미지 분석, LLM)을 돌려서 Spring Boot와 통신하는 용도.
    - **API Documentation:** **Swagger (OpenAPI)**. 프론트랑 협업하려면 필수!
    - Puppeteer/Playwright : 브라우저를 사람이 직접 마우스와 키보드로 조작하는 대신, 코드로 제어하는 엔진.
        - 용도 : 
        현재 화면이 어떤 상태인지(이미지 위주인지, 팝업이 떴는지) Vision AI가 분석할 수 있도록 **실시간 화면을 이미지로 저장**합니다.
    - MCP : AI 모델(Claude, GPT 등)이 브라우저 제어 도구들을 **안전하고 체계적으로 호출**할 수 있게 해주는 연결 규격
        - 용도 : AI가 "브라우저 열기", "검색어 입력", "스크롤 내리기" 같은 기능들을 마치 자신의 원래 능력인 것처럼 가져다 쓸 수 있게 합니다.
- **AI Engine (핵심 두뇌)**
    - **Vision AI (눈):** **Google Cloud Vision API** 또는 **GPT-4o (Vision)**. 상품 이미지를 분석해서 "빨간색 체크무늬 긴팔 셔츠입니다"라고 텍스트로 바꿔줌.
        - [Google Cloud Vision API](https://cloud.google.com/vision?hl=ko)
    - **Conversational AI (입):** **OpenAI API (GPT-4)**. "이거 세탁 어떻게 해?"라고 물으면 상세페이지를 읽어서 대답해 주는 챗봇.
- Database
    - **MySQL (RDBMS):**
        - **용도:** 회원 정보, 상품 목록, 주문 내역 등 **구조화된 핵심 데이터**를 저장합니다.
        - **선정 이유:** 데이터의 정확성이 중요한 결제/주문 시스템에서 가장 안정적이고 널리 쓰이는 표준입니다.
    - **Redis (In-memory DB):**
        - **용도:** 사용자와 AI가 나누는 **실시간 대화의 맥락(Context)** 저장 및 자주 찾는 데이터 임시 저장(캐싱).
        - **선정 이유:** 속도가 매우 빨라서, 대화가 끊기지 않고 자연스럽게 이어지도록 도와줍니다.
    - **Vector DB (예: Pinecone 또는 Milvus) (선택 사항):**
        - **용도:** 상품의 특징(예: "따뜻한 느낌의 니트")을 AI가 이해할 수 있는 숫자 형태(벡터)로 저장.
        - **선정 이유:** 시각장애인이 "포근한 옷 찾아줘"라고 추상적으로 말해도 AI가 찰떡같이 검색하기 위해 필요합니다.
- Infrastructure
    - **AWS (Amazon Web Services):**
        - **EC2 (가상 서버):** 서비스를 24시간 구동하는 컴퓨터.
        - **S3 (파일 저장소):** 상품 이미지 파일과 음성 녹음 파일을 저장하는 무제한 클라우드 폴더.
    - **Docker:**
        - **용도:** 개발한 프로그램을 어떤 컴퓨터에서든 똑같이 실행되도록 포장하는 **'컨테이너'** 기술.
        - **선정 이유:** "내 컴퓨터에선 되는데 네 컴퓨터에선 안 돼"라는 문제를 방지합니다.
    - **GitHub Actions (CI/CD):**
        - **용도:** 코드를 저장하면 자동으로 테스트하고 서버에 배포해 주는 **'자동화 로봇'**.
        - **선정 이유:** 개발자가 일일이 수동으로 서버에 올리는 시간을 줄여 효율을 높입니다.
    - **SSL (HTTPS):**
        - **용도:** 보안 인증서.
        - **선정 이유:** 마이크(음성 인식) 권한을 쓰려면 웹사이트 보안(자물쇠 표시 🔒)이 필수입니다.

**4. Infrastructure/DevOps (1명)**

- **SSE (Server-Sent Events) 구성:** MCP는 보통 SSE라는 기술로 통신해. 웹 서버(백엔드)가 AI한테 데이터를 실시간으로 밀어넣어 줄 수 있는 환경을 구축해야 함.
- **보안(Security) 설정:** AI가 우리 DB를 마음대로 뒤지면 안 되잖아? "이 AI는 '상품 조회'만 가능하고 '고객 삭제'는 불가능해"라고 **권한(Permission)**을 설정하고 방화벽을 침.
- **레이턴시(Latency) 최적화:** AI가 "잠시만요..." 하고 3초 이상 뜸 들이면 고객은 답답해함. AI와 백엔드 서버를 같은 네트워크(VPC) 안에 둬서 통신 속도를 빛의 속도로 만듦.
1. AI (2명)
    - **시스템 프롬프트 설계:** "너는 시각장애인 쇼핑 비서야. 사용자가 옷을 찾으면, 백엔드 친구가 준 `searchProduct` 도구를 사용해서 데이터를 찾아와."라고 AI를 교육시킴.
    - **MCP 클라이언트 연결:** 프론트엔드(또는 중계 서버)에서 돌아가는 AI 모델이 백엔드의 MCP 서버와 **실시간으로 통신**하도록 연결 고리를 만듦.
    - **응답 변환:** AI가 MCP를 통해 가져온 딱딱한 데이터(JSON)를 "고객님, 빨간색 니트가 3개 있어요."라는 **따뜻한 말(자연어)**로 바꾸도록 튜닝함.

**6. 리스크 관리 및 기대효과 (Risk & Impact)**

6.1 잠재적 리스크 및 해결 방안

- 기술적 리스크: 할루시네이션(환각 현상) 및 오동작
    
    문제: AI가 상품 가격이나 옵션을 잘못 읽어 엉뚱한 물건을 결제할 가능성.
    
    해결: 결제 전 최종 단계에서 'Double Check' 프로세스 도입(상품명, 가격, 옵션을 음성으로 다시 확인받는 과정 필수화).
    
    개인정보 이슈: 주소나 전화번호를 말할 때 보안 문제.
    
    *→ Solution:* 민감 정보는 음성이 아니라 **보안 키패드(TTS 지원)**를 통해 입력받도록 UI 분리.
    
    → **보안 전용 브라우징 환경:** * 결제 단계에서는 MCP 제어를 일시 중지하고, 사용자의 음성 명령과 보안 토큰만 주고받는 암호화된 세션을 통해 개인정보 보호 강화.
